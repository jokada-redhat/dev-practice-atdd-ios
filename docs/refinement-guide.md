# リファインメントガイド: PBI から feature ファイルへ

PBI（プロダクトバックログアイテム）を受け入れテストのシナリオに落とし込むまでの流れを説明します。

## 全体の流れ

```
PBI（ビジネス要件）
  ↓  リファインメント
受け入れ条件の具体化
  ↓  シナリオ洗い出し
feature ファイル作成
  ↓  ATDD サイクル
実装・テスト・リファクタリング
```

## 1. PBI を読み解く

まず PBI の受け入れ条件を確認します。

例として「会員の貸出冊数に上限を設ける」PBI の受け入れ条件:

> - 会員は最大 3 冊まで同時に借りることができる
> - 3 冊借りている状態で新たに借りようとした場合、借りられない旨が伝わること
> - 返却すれば、再び上限まで借りることができる

## 2. 前提となる振る舞いを洗い出す

受け入れ条件を実現するために、そもそもどんな振る舞いが必要かを考えます。

「貸出上限」を検証するには、まず「借りる」と「返す」ができなければなりません。これらが未実装であれば、先にシナリオとして定義する必要があります。

```
PBI: 貸出上限（3冊）
├── 前提: 書籍を借りられること
├── 前提: 書籍を返却できること
└── 本題: 上限チェックが機能すること
```

このように、1 つの PBI から複数の振る舞いが導出されます。

## 3. シナリオを洗い出す

各振る舞いについて、正常系と異常系のシナリオを考えます。

### 「借りる」の振る舞い

| 条件 | 期待結果 | シナリオ名 |
|------|---------|-----------|
| 書籍が貸出可能 | 借りられる | 会員が書籍を借りる |
| 書籍が既に貸出中 | 借りられない | 既に借りられている書籍は借りられない |
| 会員が存在しない | エラー | 存在しない会員IDで書籍を借りようとする |
| 書籍が存在しない | エラー | 存在しない書籍を借りようとする |

### 「返す」の振る舞い

| 条件 | 期待結果 | シナリオ名 |
|------|---------|-----------|
| 自分が借りている書籍 | 返却できる | 会員が書籍を返却する |
| 借りていない書籍 | エラー | 借りていない書籍は返却できない |
| 他の会員が借りている書籍 | エラー | 別の会員が借りている書籍は返却できない |
| 借りて返すと再び貸出可能 | 貸出可能に戻る | 書籍を借りて返却すると再び貸出可能になる |

### 「上限」の振る舞い

| 条件 | 期待結果 | シナリオ名 |
|------|---------|-----------|
| 2冊借りていて3冊目を借りる | 借りられる（上限内） | 貸出上限ぴったりまで借りられる |
| 3冊借りていて4冊目を借りる | 借りられない | 貸出上限に達している場合は借りられない |
| 3冊→1冊返却→別の本を借りる | 借りられる | 返却すれば再び借りられる |

## 4. シナリオの優先順位を決める

すべてを一度に実装する必要はありません。以下の順で進めるのが自然です:

**第1ステップ: 基本の貸出**
- 会員が書籍を借りる（正常系）
- 既に借りられている書籍は借りられない

**第2ステップ: 返却**
- 会員が書籍を返却する
- 借りていない書籍は返却できない

**第3ステップ: 上限チェック**
- 貸出上限ぴったりまで借りられる
- 貸出上限に達している場合は借りられない
- 返却すれば再び借りられる

**第4ステップ: エッジケース**
- 存在しない会員/書籍のエラー
- 別の会員が借りている書籍の返却エラー

前のステップのシナリオが GREEN になってから、次のステップに進みます。

## 5. feature ファイルに書く

洗い出したシナリオを `.feature` ファイルに記述します。

### 書くときのチェックリスト

- [ ] **ビジネス言語で書いているか**: 技術用語（API、DB、ID）ではなく、業務の言葉を使う
- [ ] **Given はセットアップか**: 状態の「用意」であって「検証」ではない
- [ ] **When は 1 アクションか**: ユーザーの操作は 1 つに絞る
- [ ] **Then は期待結果か**: 「こうなるべき」が明確か
- [ ] **各シナリオは独立しているか**: 他のシナリオの結果に依存していないか
- [ ] **自明な検証を省いているか**: 前の行から分かることを繰り返していないか
- [ ] **テスト済みの振る舞いを宣言的に使っているか**: 手順の繰り返しではなく状態の宣言にしているか

### 例: 上限チェックのシナリオ

```gherkin
Scenario: 貸出上限に達している場合は借りられない
  Given 会員 "山田太郎" (ID: "DA-8821") が登録されている
  And 会員 "DA-8821" が 3 冊借りている状態である
  And 書籍 "Neuromancer" が登録されている
  When 会員 "DA-8821" が書籍 "Neuromancer" を借りようとする
  Then エラーメッセージ "貸出上限（3冊）に達しています" が返される
```

- `3 冊借りている状態である` は宣言的な Given（借りる手順は省略）
- `借りようとする` は失敗が期待されるアクション（`借りる` は成功前提）
- エラーメッセージでユーザーに何が起きたか伝わる

## 6. ATDD サイクルに入る

feature ファイルができたら、ATDD サイクル（RED → GREEN → REFACTOR）で実装を進めます。詳細は [ATDD ガイド](atdd-guide.md) と [貸出上限ウォークスルー](borrowing-limit-walkthrough.md) を参照してください。

## まとめ

```
PBI の受け入れ条件を読む
  → 前提となる振る舞いを洗い出す
    → 正常系・異常系のシナリオを列挙する
      → 優先順位をつけて段階的に feature ファイルに書く
        → ATDD サイクルで実装する
```

PBI は「何を達成したいか」だけを書きます。「どうテストするか」「どう実装するか」はリファインメントと ATDD サイクルの中で具体化していきます。
